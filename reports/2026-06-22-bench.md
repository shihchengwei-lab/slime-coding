# Bench — does Slime suppress garbage on a fixture with architectural room? — 2026-06-22

The 06-21 BvC and the 06-22 model-class report both ran on `cli-notes`, a
115-line single-file CLI where a "garbage" outcome (registry pile-up for a
one-variant requirement) has nowhere to land — the literal dispatch surface
is one if-branch in one function. The 06-22 walk-back commit `5a62fb2`
flagged this as a methodological gap: the matrix's null on Haiku/Sonnet
can't be read as "Slime doesn't help weak models" if the failure mode the
gates target isn't present in the baseline either.

This run closes that gap with a new fixture, `csv-tsv-pipeline`, that has
the architectural room cli-notes lacked.

## Fixture: csv-tsv-pipeline

A multi-module Python package: `pipeline/cli.py` + `pipeline/readers/csv_reader.py`
+ `pipeline/writers/json_writer.py` + `tests/test_pipeline.py`. ~115 LOC
total, but the shape — a `readers/` subpackage with one reader today —
explicitly invites a follow-up format-extensibility task to either match
minimally (one more file under `readers/`) or balloon into a registry,
strategy, base-class hierarchy.

**Task B1**: `Add support for TSV input. We'll likely add more input formats
(XML, YAML) later. Keep python3 -m pytest -q green.`

Two reference solutions ship with the fixture under `references/`:

- `add-tsv-minimal.diff`: one new `tsv_reader.py`, one if-branch in cli.py.
  Score: M1=0 / M3=0 / M5=0 / M4=1 / M6=24.
- `add-tsv-garbage.diff`: `Reader` ABC, `CsvReader` / `TsvReader` concretes,
  `ReaderRegistry` plugin pattern, 3 new files, comments naming
  XML/YAML/Parquet as "future readers". Score: M1=4 / M3=1 / M5=6 /
  M4=3 / M6=92.

The references calibrate `experiments/score.py` on this fixture and serve
as the comparison anchor for the model cells.

## Matrix and runner

3 models × 2 conditions × N=3 = **18 cells**. Same one-shot `claude -p`
modality as the 06-21 BvC and 06-22 model-class runs. Same 590 s timeout.

- Haiku 4.5, Sonnet 4.6, Opus 4.7
- baseline (naked fixture, no L0 prose, no hooks) vs hooked-slime
- Runner: `experiments/run-bench.sh`, mirrors `run-bvc.sh` with two diffs:
  fixture is `csv-tsv-pipeline` instead of `cli-notes`, and the single
  task B1 (not E1/E2/E3).

## Scoring

Two layers:

1. `experiments/score.py --sweep <runs-root>` — six regex metrics. Calibrated
   sharp on the two reference solutions (5/6 strictly minimal < garbage; M2
   ties because the garbage's class methods aren't column-0 `def`).
2. Manual grep for `^\+[A-Z_]+\s*=\s*\{` — catches the dict-based dispatch
   form score.py's class-only M1 misses. The reference garbage uses class
   `ReaderRegistry`; the model cells turn out to prefer the dict form, and
   ignoring it would under-report the matrix.

## What the numbers say

1. **No model built the class-hierarchy garbage the reference targets.**
   M1 (new class), M3 (Registry/Strategy/Factory/...), M5 (future/later words)
   are 0 across all 18 cells. The class-ABC + concrete-Reader + Registry
   reference exists in `references/add-tsv-garbage.diff` but no model
   produced it.

2. **Models DID build the dict-dispatch form of registry.** 4/18 cells
   landed a module-level `READERS = {".csv": …, ".tsv": …}` (or `_READERS`)
   table — the pythonic version of the same speculative-extensibility move.
   This form scores 0 on M1/M3/M5 but is the same architectural decision.

3. **Slime does not reduce dict-registry incidence; on Opus it increases it
   in this run.** Opus baseline 1/3 → Opus hooked 2/3. Sonnet baseline 1/3
   → Sonnet hooked 0/3, but Sonnet hooked also produced 2/3 cells with no
   implementation at all — the model can't build a registry in a cell that
   never writes any code.

4. **Sonnet pays a real cost under the hooks.** baseline 0/3 cells were
   empty, hooked 2/3 were. The Sonnet probe earlier in the day (cli-notes)
   showed the same shape: the agent spends the 590 s budget on corridor
   authoring + repo navigation before reaching the implementation. The
   csv-tsv-pipeline fixture reproduces this at a higher rate (2/3 vs
   cli-notes's 2/9).

5. **Haiku is approximately unaffected.** 1/3 hooked empty is at the edge
   of N=3 noise; no other cells differ from baseline in shape or size.

## Direct answer to "does Slime work?" — on this fixture

**No measurable positive effect from Slime in this matrix, plus a real cost
on Sonnet.** Stated narrowly:

- The thing Slime was designed to suppress (speculative dispatch tables for
  a one-variant requirement) appears in 4/18 cells in this matrix —
  exclusively as a dict registry, not the class-ABC form. The Slime-on cells
  have *more* of these than the Slime-off cells on Opus (2 vs 1), the same on
  Haiku (0 vs 0), and fewer on Sonnet (0 vs 1) — but Sonnet's hooked column
  only produced 1 non-empty cell to count, which dominates that delta.
- The wall-clock cost the corridor flow imposes on Sonnet is the same shape
  the cli-notes run flagged in commit 7a4c058: 2/3 cells finished with no
  implementation. Opus has the deliberation budget to absorb it; Sonnet at
  this configuration doesn't.

## What this does NOT prove

- **N=3 per cell, one fixture family, one bait.** Any single delta is at
  noise level. The *direction* of the table (Slime doesn't help, sometimes
  hurts) is what's load-bearing here, not any individual cell.
- **"Slime is broken / useless / harmful."** Not what the data supports.
  The gates' separately-measured value as a backstop on git facts (new deps,
  hallucinated references, failing checks at Stop) did not fire in this
  matrix because the agents didn't trigger them. That value is real and
  measured elsewhere (06-18 mechanism verification, the L2 acceptance tests
  in `tests/test.sh`).
- **A class-ABC garbage outcome.** The garbage reference uses one. Models
  did not produce one in this matrix. That doesn't mean they never would on
  a different fixture or under different prompting; it means this run can't
  evaluate Slime's effect on a failure mode it didn't elicit.
- **"AI has improved and doesn't write garbage anymore."** The dict-registry
  form appears in 4/18 cells and is the same architectural move the
  reference grabs at — just more pythonic. The visible class hierarchy
  garbage of the reference is rarer than score.py's class-only M1 made it
  look across the prior reports.

## score.py — what the calibration missed

The reference garbage solution uses ABCs + concretes + a class-based
`ReaderRegistry`. score.py's M1 (`^+class`) and M3 (Registry/Strategy/...
in added lines) catch this cleanly: garbage scores 4/1/6 vs minimal
0/0/0 on M1/M3/M5.

The model cells turned out to prefer the dict form (`READERS = {...}`),
which scores 0 on all three flagged metrics. M3 misses it because
"READERS" isn't on the keyword list, and the variable name `READERS` /
`_READERS` is domain-specific to the fixture, not generic. M1 misses it
because there's no class.

In retrospect, the score.py rubric would have benefited from an M7
"module-level uppercase dict literal" check (`^\+[A-Z_]+\s*=\s*\{`). I
ran this as a one-shot grep for this report instead of editing score.py
(score.py is already committed and the calibration claim is on
class-form garbage; widening it post hoc would invalidate the recalibration
on prior reports). A future fixture run should bake this check into the
scorer as M7 alongside fresh calibration on a new minimal/garbage pair.

## Scope / caveats

One fixture (Python `csv-tsv-pipeline`), one bait family (format
extensibility), N=3 per (model × cond) — 18 cells of evidence here, with
two reference solutions calibrating score.py on the fixture. No
non-Python fixture, no non-extensibility bait, no other modality, no
interactive sessions. The dict-registry detection was added post-hoc as a
manual grep, not as a calibrated metric — a future run should fold it in
as M7 with fresh calibration.

No "validated / proven / production-ready" language is warranted. README
stays experimental.
