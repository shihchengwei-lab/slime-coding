# BvC — does the hooked stack add anything beyond prompt-only? — 2026-06-21

Follow-up to `2026-06-18-extensibility-3runs.md`, which established **A baseline
vs B prompt-only** on the speculative-extensibility axis (E1 export / E2 stats /
E3 sort, "we'll add more X later"). The open question it left (and that
`2026-06-18-condition-c.md` flagged as "out of reach"): does the **hooked**
condition **C** add anything *beyond* the prose of B?

This run answers it with a **fully automated** condition C — the thing the prior
condition-c report said "needs the hooks firing inside the agent-under-test's
session — out of reach for spawned sub-agents here." It is no longer out of
reach (see *Feasibility* below).

**Headline: at matched modality, C does not measurably beat B on this axis
(N=3).** The prose carries the suppression; the gates add no further cut here —
because speculative abstraction is not one of the git facts the L2 gates target.

## Feasibility — the automated C that was "out of reach"

The blocker was never an inability to control hooks; it was the assumption that
the agent-under-test must be a *sub-agent of the host session* (whose tool calls
the host's hooks can't intercept). The fix is to run the agent as a **separate,
cell-rooted, non-interactive process**:

```
cd <cell> && claude -p "<task>" --setting-sources project,local \
  --permission-mode bypassPermissions --output-format stream-json --verbose
```

`--setting-sources project,local` does double duty: it loads the cell's project
hooks (installed by `install.sh` into `<cell>/.claude/settings.json`) for C, and
it **excludes the host owner's `~/.claude`**, so the B arm genuinely runs with no
hooks. Verified end-to-end; wrapped as one command in `experiments/run-bvc.sh`.
OAuth/CLI only — no billed API.

### Harness bug found and fixed first (would have voided the comparison)

`slime-bench` seeded the L0 discipline as a bare `CLAUDE.slime.md`. Claude Code
auto-loads `CLAUDE.md`, **not** `CLAUDE.slime.md`. An empirical probe (a
spontaneous-directive file the agent only obeys if auto-loaded) confirmed the
bare file is **inert**: with it, an unrelated prompt ignored the directive; in
`CLAUDE.md`, the directive fired. Left unfixed, automated B/C would have lost
their defining prose and **collapsed onto baseline A**. Fixed so both arms seed
the discipline into `CLAUDE.md`; `tests/test-bench.sh` now asserts it (12/12).

## Why B was re-run at the same modality (overriding "reuse historical B")

The 2026-06-18 A/B runs were **host-driven and interactive**. The automated C
here is **one-shot `claude -p`**. Those are different modalities, and they are
**not interchangeable** on this axis — so comparing one-shot C to interactive B
would confound *hooks* with *modality*. Re-running B one-shot (same harness) is
what makes the B-vs-C delta isolate the hooks. This is a deliberate departure
from the task's "do not re-run B".

The departure was load-bearing: on E3, interactive B suppressed the registry
0/3, but **one-shot B leaks it 2/3**. Had I trusted the historical B, I would
have reported a spurious "C is much worse than B (0/3 → 3/3)". Most of that jump
is modality, not hooks.

## Verdict — does C differ from B?

**No measurable difference between B and C at N=3 (signal, not proof).** On the
speculative-abstraction axis, the prose does the work; the hooked gates add no
further reduction. This is the expected shape, for a concrete reason: the L2
gates fire on **git facts** (a new dependency, a hallucinated reference, a
failing check, a missing corridor) — a `SORT_KEYS` dict is none of those, so the
gates **cannot** catch it by design. C therefore can't beat B on *this* axis;
the gates' value (shown in `2026-06-18-condition-c.md`) is as a backstop on the
facts they do target, not a multiplier on premature abstraction.

Consistent with the prior finding: L0 prose changes the common case; L2 gates
catch the non-compliant case on unambiguous facts. They cover different cases,
and this axis is L0's, not L2's.

## Secondary finding — modality weakens YAGNI suppression on the lightest bait

E3's `SORT_KEYS` is a one-line dict — the cheapest of the three baits to
rationalise as "idiomatic", and the extensibility report already noted the
lighter the abstraction the more it leaks (E2's one-key dict). Under the
**one-shot** modality, that leak widens: both B (2/3) and C (3/3) build the
registry on E3, where interactive B suppressed it (0/3). The one-shot agent
authors a corridor and sees the injected PRUNED context (hooks fire — see
isolation) but deliberates less about pruning the abstraction. This is a
property of the run modality, not of the hooks.

## Isolation verification (the result's credibility)

Per-arm hook behaviour, counted from the agents' own stream-json transcripts
(the prune-inject injection header "These designs were already rejected"):

- **C fired:** 9/9 hooked-slime cells injected the PRUNED context (count = 1 each).
- **B silent:** 0/9 prompt-only cells; no prompt-only cell has a
  `.claude/settings.json` at all.

The contamination guard holds — the comparison is not the ponytail-class bug
where a SessionStart hook fires on every arm. C and B are distinguishable in
hook behaviour exactly as intended.

## Scope / caveats

One fixture family (Python `cli-notes`, export/stat/sort registry axis), one
model class, **N=3**, one bait phrasing. The C-vs-B null is on the
speculative-abstraction axis only; it says nothing about the gates' backstop
value on deps/hallucinated-refs/failing-checks (measured elsewhere). The
historical A/B columns are interactive-modality and shown only for context — not
the comparison. No "validated / proven / production-ready" language is
warranted. README stays experimental.
